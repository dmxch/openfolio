## Verifizierte Korrektheit

**Drawdown-Rekonstruktion (16d4b6b)**
- `_portfolio_drawdown`/`_bucket_drawdown` korrekt zu `_indexed_drawdown` verschmolzen; oeffentliche Signatur `get_max_drawdown` unveraendert, alle 4 Caller (buckets.py:446, performance.py:76, external_v1.py:349/745) kompatibel.
- Output-Schema (`peak_value_chf`, `trough_value_chf`, `current_vs_peak_pct`, `drawdown_brake_active`, ...) identisch — kein Frontend-Bruch.
- **PE-/Cash-/Pension-Ausschluss haelt**: Bucket-Filter setzt nur `Position.bucket_id`; history_service schliesst PE/Cash/Pension in `tradable_positions` (123) und `calc_portfolio_value` (289-296) weiterhin aus. Selbst wenn ein User-Bucket eine PE-Position enthielte, wird sie nicht bewertet.
- **System-Buckets (PE/Immobilien)**: Der Worker `check_bucket_drawdown_brakes` iteriert ausschliesslich `BucketKind.user` (bucket_drawdown_service.py:84-87) — System-Buckets werden nie an den Drawdown gefuettert. HEILIGE Regeln 4/6 gewahrt.
- **`period="all"`-Pfad** (Worker): `_indexed_drawdown` setzt `hist_start=2000-01-01`, history_service kuerzt intern auf echte Inception (75-83) — keine synthetische Vorhistorie.
- **Transaktions-Query NICHT bucket-gefiltert** (history_service:69): bewusst unkritisch — out-of-bucket-Holdings landen in `current_holdings`, werden aber in `calc_portfolio_value` via `positions.get(pid) is None → continue` (286-288) verworfen. `cashflows_by_date` (89/105/107) ist toter Code (nie konsumiert), daher kein Leak in den Index. Der bucket-lokale `value_before_cf`-vs-`prev_value`-Mechanismus rechnet korrekt nur auf In-Bucket-Holdings.

**GBX/Pence-Fix (6c461b2)**
- `_pence_divisor` (cache_service.py:54-64) gibt 1.0 fuer Nicht-Pence-`.L` (EIMI.L=USD wird NICHT geteilt) und 1.0 bei unbekannter/None-Quote-Waehrung (kein Raten). Korrekt.
- Divisor skaliert nur `_price_series` (history_service:231-233); FX laeuft danach ueber die echte Quote-Waehrung (`_resolved_currency`→GBP), d.h. `GBPCHF=X` statt der falschen `pos.currency`. `info["currency"]`-Override (171-175) nur fuer `.L`-Ticker.
- yfinance fast_info (`_quote_currency`) ist in **einem** `asyncio.to_thread` gekapselt (166-168); Resultat 7-Tage-gecacht (cache_service:50) → kein Burst/429-Risiko im Steady-State, nur 1 Call/Ticker beim Cold-Cache. Projekt-Regel 7 erfuellt.
- **Index-Mathematik unveraendert**: Der Sub-Period-Return ist ratio-basiert; ein konstanter Divisor kuerzt sich im `portfolio_indexed` ohnehin heraus. Der Fix korrigiert nur den nominalen `value` (= raw_value des Drawdowns). Keine HEILIGE-Regel-1/11-Datei beruehrt.

**6m-Periode (3c9c0c5)**
- Pattern `^(1m|3m|6m|ytd|1y|all)$` + `elif period == "6m": start = today - 182d` (external_v1.py:301-302) konsistent mit dem schon vorhandenen `_VALID_PERIODS`/`_period_start` im drawdown_service (40, 50). 182<365 → kein Downsample-Edge. User-scoped, rate-limited.
